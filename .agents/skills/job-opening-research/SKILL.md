# Job Opening Research

Find the openings a candidate should actually see — not just the ones easy keyword queries surface. The workflow is **discovery as a system**: thesis → universe → sources → log → self-audit, with every phase leaving an auditable artifact. `references/search-workflow.md` is normative — read it before running; it defines the artifact schemas, the ATS source matrix with URL patterns, the fallback ladder, and stopping criteria. Also read `references/search-source-registry.yaml` before building the company universe/source plan; use relevant curated company sets and ATS discovery patterns as seeds, verifying every URL at runtime. `references/recruiting-sheet-schema.md` governs store writes.

Work in the recruiting repo root; resolve the active search project via `active-project.json` (`references/project-structure.md`) — artifacts live under `<project>/targets/`.

## Why this structure exists

A previous live run missed directly relevant openings because discovery relied on the user's seed companies and title-keyword queries, and left no trace of what was considered. Result: misses were undiagnosable (unseen? seen-and-skipped?) and unrepeatable. The five artifacts fix both: they force breadth *before* searching and auditability *after*.

## Your judgment vs scripts

You (the agent) own every decision: interpreting the profile into a thesis, choosing which adjacent companies belong in the universe and why, picking sources, judging each posting, deciding when coverage is sufficient. `scripts/validate_research_artifacts.py` only checks mechanics afterward — artifacts present, schemas valid, every universe company attempted, every failure carrying its fallback trace. It will never tell you which company to search; that reasoning is yours and must be written down as rationale.

## The five phases (artifacts mandatory)

1. **Opportunity thesis** → `targets/opportunity-thesis.md`. Role *families* (strategy/ops, BD/partnerships, GTM, product strategy, chief of staff, ecosystem, AI deployment...), seniority band, hard constraints, domain buckets, evidence-backed strengths/gaps. Confirm direction with the user if the project is new; otherwise derive from profile + project.json + scoring config. If `candidate-profile.md`/`evidence-map.md` do not exist yet, search still proceeds from declared project targets, LinkedIn/source hints, and user instructions; label profile-dependent fit assumptions as provisional and recommend `prep` after search to build the evidence-backed profile.
2. **Company universe** → `targets/company-universe.json`. Seeds are a floor, not a ceiling. **Treat every user-named company as an *example of a market map*, not only a literal target**: for each seed, infer its *archetype* (scale/maturity, business model, product category, talent pool, location market), then expand that archetype into peers — competitors, same-talent-pool employers, adjacent product categories, ecosystem partners, funded entrants, and location-relevant employers. Fix the relationship *types*, never specific company names. Name-brand seeds (large platforms, hyperscalers, frontier labs, category leaders) are the ones most often left un-expanded — a universe that searches only the seed plus one or two neighbors is a coverage defect, not a focused search; the same generalization you already apply to a category seed like `fintech startup` applies to a name-brand seed like `Google` or `Meta`. Use relevant `search-source-registry.yaml` company sets as seed/adjacency hints when they match the thesis, never as a mandatory universal list. Every company carries a rationale tied to the thesis, every exclusion carries a reason. **Per-archetype floor**: if a seed's archetype yields fewer than ~5 searched companies, either expand it or record why expansion is inappropriate (in `excluded`, or a bucket `expansion_note`). Default scale: seeds + 25–50 adjacent across 5–8 buckets. Before saving, run one **omissions self-critique** — "given these seeds, role families, and locations, what obvious peer employers are missing?" — and add each with rationale or exclude it with a reason (carry the result into `coverage-audit.md`).
3. **Source plan** → `targets/source-plan.json`. Per company: trusted source path in trust order (ATS page → official careers → web discovery → LinkedIn Jobs mandatory pass under the login-handoff contract → boards as signal). **Self-hosted careers sites** — many employers publish NO third-party ATS board; that absence is *expected*, never a `failed_capture` on its own, and is *detected* (ATS-slug probes come back empty), not assumed from a company's name. When there is no ATS board, go to the official careers site: consult the registry's `self_hosted_careers` section — a curated set of examples (each with its careers search endpoint, `posting_url` pattern, and `render` type; verify at runtime) — and use that adapter; for a company **not** in the registry, web-discover its careers site and apply the same method (careers JSON API / search endpoint → enumerate → capture each posting's own `posting_url` → parse its JD). Client-rendered (`render: js`) sites need a browser runtime (install Playwright if none is present); server-rendered / JSON-API sites do not. Try ATS URL patterns from the reference and registry directly — Greenhouse (`boards.greenhouse.io`, `job-boards[.eu].greenhouse.io`), Lever, Ashby, Workable, SmartRecruiters, Workday, plus the registry's extended families (Recruitee, Teamtailor, Personio, BambooHR, Breezy HR, Pinpoint, Rippling, iCIMS/JibeApply, Jobvite, SuccessFactors, Comeet — most expose zero-auth per-tenant feeds). Prefer the branded careers URL as `source_url` when one exists (registry `branded_url_rule`), and check the registry's `posting_expiry_signals` (e.g. Greenhouse `?error=true`, "no longer available" text, shell pages) before persisting. Aggregator boards/feeds in the registry's `job_boards_and_aggregators` are discovery signals — verify and capture at the company/ATS source. On structured boards, ENUMERATE all postings for the target location and judge each (log a `board_enumeration` query with the total) — never keyword-filter a board at the source. **Fallback ladder is enforced**: `failed_capture` is legal only after ≥3 distinct source types, recorded in order. A JS-blocked page or an empty static HTML is a fallback trigger, never a stopping point.
4. **Candidate log** → `targets/research-log.json`. Every candidate seen anywhere — `kept/skipped/failed_capture` + reason code; queries with `results_seen` (≤1 results = logged coverage signal, not evidence of absence). Capture per posting: company, exact title, location, remote policy, seniority, must-have vs nice-to-have, 2–3 sentence JD summary, URL, date. Never enter credentials or bypass access controls yourself. The LinkedIn Jobs pass runs LAST, after phases 1-3 and all non-login sources; declare it blocked only after navigating to linkedin.com/jobs and recording the observed state in the log (`jobs_page_ok` = proceed). If blockage is observed, finish and persist everything else first, then STOP with the `APPROVAL_REQUIRED` handoff from `references/search-workflow.md` — the user signs in, and the rerun executes only the LinkedIn increment. Snapshot JD text to `targets/jobs/<job_id>.json`.
5. **Coverage self-audit** → `targets/coverage-audit.md`. Thin buckets and why, dead-ended fallbacks, low-coverage signals, prioritized next searches. Your user-facing summary quotes this — never report done over unexplained thin buckets.

Stopping criteria are coverage goals (every company attempted; every family×location queried; marginal yield <2 new candidates over last 3 queries), not query quotas — budgets in the reference are defaults the thesis may override, and running out of budget is reported, not silently absorbed.

## Parallel capture (universes >~15 companies)

Shard Phase 4 across **3–6 parallel subagents** per the protocol in `references/search-workflow.md` ("Parallel capture with subagents"): the lead builds phases 1–3 centrally, hands each subagent a bucket-based company shard + thesis + registry rules, and each subagent writes only `targets/research-log.parts/<shard>.json` (full log schema, JD snapshots included) — no DB writes, no LinkedIn pass, no user interaction from inside a shard. The lead merges parts into `research-log.json`, dedupes by canonical URL, runs the LinkedIn pass last, then validates and persists once. A failed shard is rerun and reported, never silently dropped. Universes under ~10 companies run inline without subagents.

## Persisting rows

Write every intermediate JSON below **inside `<project>/data/`** (guaranteed writable) — never `/tmp`, `C:\tmp`, or an absolute OS path; on Windows those fail (FileNotFoundError/PermissionError) and a broken persist step silently drops the rows you captured. Persist **all** captured rows (one upsert, or resumable batches kept under `<project>/data/`) — a partial write is how a run ends up with far fewer rows than it found.

1. Ensure store exists: `python scripts/init_db.py`; check project.json for any external sheet copy.
2. Canonicalize: `python scripts/normalize_job_url.py --json <project>/data/rows.json > <project>/data/rows_normalized.json`.
3. Validate: `python scripts/validate_job_rows.py <project>/data/rows_normalized.json --existing <project>/data/job_list.csv` — never write invalid rows.
4. Write: `python scripts/upsert_rows.py job_list <project>/data/rows_normalized.json` (transactional upsert, read-after-write, views regenerated).
5. Mechanics check: `python scripts/validate_research_artifacts.py <project>` — fix every FAIL before reporting (this FAILS the run if any declared seed/preference company is absent from the universe or was never attempted).

## Reporting

Summarize: found vs saved vs skipped (with reason distribution), universe coverage (attempted/total companies per bucket), remaining failed_captures with next steps, and the self-audit's follow-up priorities. Distinguish page-captured facts from your judgments. Suggest `target-job-group-strategy` when coverage is adequate — or another research pass targeting the audit's thin spots when it isn't.

**Every user-named (preference/seed) company gets an explicit line in the summary** — `<Company>: N kept / M skipped / failed_capture (why) / excluded (why)`. Never let a named company vanish from the report: a seed that produced zero saved rows is a *reported outcome with its reason* (e.g. "<Company>: JS careers page, no browser runtime available → failed_capture; retry with the browser connector"), not a silent absence. If a preference company shows 0 saved rows, say so at the top so the user can decide to rerun rather than discovering it by counting rows.
